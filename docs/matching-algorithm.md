# AIマッチングアルゴリズム

## 概要

本システムは、薬剤師の希望と薬局の募集を自動的にマッチングします。**候補者が少ない薬局を優先的に処理**することで、不足薬局をゼロに近づけることを最優先とし、同時に距離・応募回数・評価を総合的に考慮した最適な割り当てを実現します。

---

## アルゴリズムの3つの目的

| 優先度 | 目的 | 説明 |
|--------|------|------|
| **1** | **不足薬局をゼロに** | 候補者が少ない薬局から優先的に処理し、できるだけ多くの薬局に薬剤師を割り当てる |
| **2** | **距離を考慮** | 薬剤師の住所と薬局の距離を考慮し、通いやすい組み合わせを優先（重み40%） |
| **3** | **公平性を保つ** | 応募回数が少ない薬剤師を優先し、特定の薬剤師に偏らないようにする（重み30%） |

評価（重み30%）も考慮し、総合的に最適な組み合わせを選択します。

---

## 処理の流れ

### ステップ1: データ準備
確定済みシフトを除外し、その日の未確定の希望・募集のみを対象にします。

### ステップ2: 互換性判定
各薬剤師と薬局の組み合わせについて、以下をチェックします：
- **時間の互換性**: 薬剤師の希望時間が薬局の募集時間を100%カバーしているか
- **NGリスト**: 相互にNG設定されていないか

### ステップ3: スコア計算
互換性がある組み合わせに対して、総合スコアを計算します：
```
総合スコア = (距離 × 40%) + (公平性 × 30%) + (評価 × 30%)
```

### ステップ4: 最適化マッチング（核心部分）
**候補者が少ない薬局から順に処理**します。これが不足最小化の鍵です：

1. 各薬局の候補者数をカウント
2. 候補者が少ない順に薬局を並べ替え
3. 各薬局に対して、スコアが最も高い薬剤師を割り当て
4. 割り当てた薬剤師は他の薬局では使用不可

この方法により、候補者が限られている薬局を優先的に埋めることができます。

### ステップ5: 再配置による改善（オプション）
不足が残った薬局に対して、以下の条件で再配置を試みます：
- 元の募集の50%以上が既に確保されている場合のみ
- 最大10回まで繰り返し

### 結果
最終的なマッチング結果を返します。手動での調整も可能です（後述）。

---

## 重要なポイント

### なぜ「候補が少ない薬局から優先」なのか？

**例**: 薬局A（候補1人）、薬局B（候補5人）がある場合

- **間違った方法**: 薬局Bから先に処理 → 薬局Aの唯一の候補が取られて不足
- **正しい方法**: 薬局Aから先に処理 → 薬局Aを確保した上で、薬局Bに残りを割り当て

候補が多い薬局は後回しにしても選択肢が残りますが、候補が少ない薬局は早く処理しないと誰もいなくなります。

### スコア計算の詳細

#### 距離スコア（40%）
薬剤師と薬局の住所を比較：
- 同じ市区町村: 0.9
- 近い地域: 0.7
- 遠い地域: 0.3
- 情報なし: 0.5

#### 公平性スコア（30%）
```
応募回数スコア = 応募回数 / 10（最大1.0）
最終スコア = (1 - 応募回数スコア) × 0.3
```
応募回数が少ないほど高スコアになります。

#### 評価スコア（30%）
```
評価スコア = (評価 / 5.0) × 0.3
```

#### 計算例
```
薬剤師A: 同じ市区町村、応募2回、評価5.0
総合スコア = (0.9 × 0.4) + ((1 - 0.2) × 0.3) + (1.0 × 0.3)
           = 0.36 + 0.24 + 0.30
           = 0.90
```

## インタラクティブマッチング

管理者が**手動で薬剤師を入れ替える**ことができる機能です。

### 基本動作

```
AIの提案:
  A薬局 → 田中さん
  B薬局 → 佐藤さん
  C薬局 → 高橋さん

管理者の判断:
  「C薬局は高橋さんより佐藤さんの方がいい」
  → ドロップダウンで変更

結果（自動再最適化）:
  A薬局 → 田中さん（変更なし）
  B薬局 → 高橋さん（自動で変更）
  C薬局 → 佐藤さん（手動で変更）(成功)
```

### 特徴

#### 固定割り当て
手動で変更した薬局は**固定**され、再最適化時も変更されません。

#### 自動再最適化
手動で変更すると、他の薬局のマッチングが**自動的に再計算**されます。

#### 候補者リスト
各薬局に割り当て可能な全薬剤師がドロップダウンで表示されます。他の薬局に割り当て済みの薬剤師も選択可能です（自動で入れ替わります）。

## 参考：処理の具体例

### シナリオ: 3月15日のマッチング

#### データ

**薬剤師の希望**:
| 薬剤師 | 希望時間 | 最寄駅 | 応募回数 | 評価 |
|--------|----------|--------|----------|------|
| 田中さん | 09:00-18:00 | 渋谷 | 2回 | 5.0 |
| 佐藤さん | 10:00-19:00 | 新宿 | 5回 | 4.0 |
| 鈴木さん | 09:00-15:00 | 池袋 | 1回 | 5.0 |
| 高橋さん | 13:00-19:00 | 渋谷 | 0回 | 5.0 |

**薬局の募集**:
| 薬局 | 募集時間 | 最寄駅 | 必要人数 |
|------|----------|--------|----------|
| A薬局（渋谷店） | 09:00-18:00 | 渋谷 | 1名 |
| B薬局（新宿店） | 10:00-17:00 | 新宿 | 1名 |
| C薬局（池袋店） | 14:00-19:00 | 池袋 | 1名 |

#### 処理手順

**1. 互換性チェック**

A薬局（09:00-18:00）と互換性がある薬剤師:
- 田中さん: OK（09:00-18:00で完全カバー）
- その他: NG（時間が合わない）

B薬局（10:00-17:00）と互換性がある薬剤師:
- 田中さん、佐藤さん、高橋さん: OK

C薬局（14:00-19:00）と互換性がある薬剤師:
- 佐藤さん、高橋さん: OK

**2. スコア計算**

A薬局 → 田中さん:
```
距離: 0.9（渋谷→渋谷）
応募: (1 - 0.2) × 0.3 = 0.24
評価: 1.0 × 0.3 = 0.30
総合: 0.90
```

B薬局の候補:
| 薬剤師 | 総合スコア |
|--------|-----------|
| 高橋さん | 0.88（最高） |
| 田中さん | 0.82 |
| 佐藤さん | 0.75 |

C薬局の候補:
| 薬剤師 | 総合スコア |
|--------|-----------|
| 高橋さん | 0.88（最高） |
| 佐藤さん | 0.67 |

**3. 最適化マッチング**

候補者数でソート:
1. A薬局: 候補1人 ← 最優先
2. C薬局: 候補2人
3. B薬局: 候補3人

割り当て実行:
```
A薬局: 田中さん（90点）を割り当て
  → 田中さんは使用済み

C薬局: 高橋さん（88点）を割り当て
  → 高橋さんは使用済み

B薬局: 佐藤さん（75点）を割り当て
  → 残った最高スコア
```

**4. 最終結果**

```
A薬局（渋谷店）: 田中さん（90点）
B薬局（新宿店）: 佐藤さん（75点）
C薬局（池袋店）: 高橋さん（88点）

不足薬局: 0件
マッチング成功率: 100%
```

## まとめ

### アルゴリズムの強み

- **不足を最小化**: 候補が少ない薬局から優先処理
- **距離を考慮**: 薬剤師が通いやすい薬局を優先
- **公平性**: 応募回数が少ない薬剤師を優先
- **柔軟性**: 手動での微調整が可能
- **タイムゾーン対応**: 月末日も正しく処理

### 処理の流れ（再掲）

```
1. データ準備 → 2. 互換性チェック → 3. スコア計算
                      ↓
4. 最適化マッチング（候補が少ない順）
                      ↓
5. 再配置による改善（不足を減らす）
                      ↓
      【マッチング完了】
```
